Nobody Told You What __init__.py Actually Does. Let Me Fix That

When most people learn Python, someone tells them: “Just put an empty __init__.py in your folder and it becomes a package."

Cool. Sure. But then you open up a real production codebase and that file has 50 lines of stuff in it and suddenly you’re like… wait, what is all this?

That’s what this article is about.

Let me give you a mental model first

Imagine you walk into a hotel. You don’t go hunting through the back office, the kitchen, and the laundry room to find what you need. You go to the front desk. The front desk handles everything and points you in the right direction.

That’s __init__.py. It's the front desk of your package.

The disaster version looks like this:

from mypackage.internal.module.deep import some_function

The nice version looks like this:

from mypackage import some_function

__init__.py is what makes the nice version possible. Now let's talk about how people actually use it.

1. Making your package nice to use (the most common one)

Let’s say your code is spread across a bunch of files — core.py, models.py, utils.py. You don't want people digging through all of that. You just want them to import your package and have everything they need right there.

So you write this in __init__.py:

from .core import Engine, run_pipeline
from .models import Model, Dataset

__all__ = ["Engine", "run_pipeline", "Model", "Dataset"]
__version__ = "2.1.0"

Now the user just does:

import mypackage
engine = mypackage.Engine()

They have no idea that Engine lives in core.py. They don't need to. That's kind of the point — their life is simpler because you did the work.

Pandas does this. FastAPI does this. Pretty much every library you actually enjoy using does this.

2. Not loading everything upfront (because nobody likes waiting)

Here’s a problem that sneaks up on you: your package grows. Now importing it takes forever because it’s loading a database module, a caching layer, an auth system… all at once, even if the user only needs one small piece.

The fix is lazy loading — you only load something when someone actually asks for it:

import importlib

__all__ = ["database", "cache", "auth"]
def __getattr__(name):
    if name in __all__:
        return importlib.import_module(f".{name}", __name__)
    raise AttributeError(f"module {__name__} has no attribute {name}")

Now database doesn't load until someone actually writes mypackage.database. Before that? It just sits there, doing nothing, not slowing anyone down.

TensorFlow does a version of this. When you’ve got millions of users, shaving half a second off import time actually matters.

3. Loading your config once, automatically

Some packages need to read a config file before they can do anything. You could make users call a setup function manually — but that’s annoying. Why not just do it when the package loads?

from .loader import load_config

CONFIG = load_config()
def get(key, default=None):
    return CONFIG.get(key, default)

User imports the package, config is already loaded, they just call:

import configpkg
db_host = configpkg.get("database_host")

No setup ritual. It just works. People love when things just work.

4. The plugin system thing (this one’s actually pretty cool)

Okay this one takes a second to wrap your head around, but stick with me.

What if you want other developers to be able to extend your package without you having to touch your own code? Like, you build an analytics system and someone else wants to add a heatmap feature. Do they fork your code? No — they register a plugin.

from .registry import PluginRegistry

registry = PluginRegistry()
def register(plugin):
    registry.register(plugin)

Now a third-party developer can just call analytics.register(their_plugin) and it hooks right in. Your core code never changes. Apache Airflow works like this — that's how it has hundreds of integrations without becoming a bloated mess.

5. The logging thing (small but important)

This one’s easy to skip but you really shouldn’t.

If you’re building a library, add this:

import logging
logger = logging.getLogger(__name__)
logger.addHandler(logging.NullHandler())

Here’s why: if your library logs something and the user hasn’t set up logging in their own app, Python prints an annoying warning at them. The NullHandler prevents that — it's basically saying "I'm logging-aware, but I'll keep quiet unless you tell me otherwise."

The requests library does this. It's one of those things you don't notice when it's done right, but you definitely notice when it's missing.

6. Putting it all together — a real example

Here’s what the __init__.py of a machine learning library might look like:

from .training.trainer import Trainer
from .inference.predictor import Predictor
from .datasets.loader import DatasetLoader
from .utils.seed import set_seed

__all__ = ["Trainer", "Predictor", "DatasetLoader", "set_seed"]
__version__ = "0.9.3"

Underneath this, there might be 50 files across multiple folders. But the person using this library just writes:

from mlsystem import Trainer, set_seed

Done. They never have to think about the folder structure. That cognitive load stays with you, the package author — and that’s exactly where it should be.

So what’s the actual takeaway here?

It’s this: __init__.py is not just a signal to Python that "this folder is a package." It's a design decision.

It’s you deciding what your users see and what they don’t. It’s you making their experience smooth instead of confusing. It’s the difference between a package that feels professional and one that makes people go “wait, where is everything?”

A few rules to carry with you:

  • Keep it lean. It should mostly be imports, not heavy logic.
  • Be deliberate about what you expose. If it’s not in __all__, treat it as private.
  • Use _internal/ folders for stuff you don't want people touching.
  • Add a __version__. It's two seconds of work and developers appreciate it.

That’s really it. Once you stop thinking of it as a required empty file and start thinking of it as your package’s front door — the whole thing clicks.
