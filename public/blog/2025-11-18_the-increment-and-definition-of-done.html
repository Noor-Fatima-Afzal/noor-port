The Increment and Definition of Done

Introduction

Imagine a restaurant where the chef considers a dish “done” when the main ingredients are cooked, never mind plating, seasoning, or temperature. The waiter considers it “done” when it reaches the table, regardless of quality. The customer expects something entirely different. Chaos and disappointment are guaranteed.

In software development, “done” is equally slippery. One developer thinks “done” means code written. Another thinks it means tested. Product Owner thinks it means deployed. Without a shared definition, teams ship half-baked features, accumulate technical debt, and disappoint stakeholders.

The Increment and Definition of Done solve this problem. The Increment is the sum of all completed work , the actual working product. The Definition of Done is the explicit quality standard that defines “complete.” Together, they create shared understanding and prevent the endless confusion of “what does done really mean?”

When teams have a clear Definition of Done and honor it religiously, quality becomes built-in, not bolted-on. When they don’t, technical debt compounds, rework multiplies, and the product degrades Sprint by Sprint.

Let me show you how to create Increments that are truly done and Definitions of Done that ensure quality.

What is the Increment?

The Increment is the sum of all Product Backlog items completed during a Sprint, plus the value of all previous Increments.

Key Characteristics:

Working Software: Not documentation, not code sitting on a branch, not “almost done”, actual functioning software.

Meets Definition of Done: Every item in the Increment has met all quality standards defined by the team.

Usable: The Product Owner could release it to users. (Whether they choose to is a separate decision.)

Cumulative: Each Increment builds on previous Increments. Sprint 16’s Increment includes everything from Sprints 1–15 plus new work from Sprint 16.

Integrated: All pieces work together as a coherent whole, not isolated components.

What “Potentially Releasable” Means

“Potentially releasable” does NOT mean:

  • Actually releasing every Sprint
  • Stakeholders forcing release
  • No additional preparation allowed

“Potentially releasable” DOES mean:

  • Technically ready to deploy
  • Meets quality standards
  • No known critical defects
  • The Product Owner could release it if they chose to

Analogy: Like a restaurant meal that’s ready to serve. The waiter (Product Owner) decides when to bring it to the table, but the kitchen ensures it’s ready whenever that decision is made.

Why this matters: Teams can’t use “we’ll fix it later” or “we’ll test before release” as excuses. Every Sprint produces releasable quality.

The Definition of Done

The Definition of Done is a formal description of the state of the Increment when it meets the quality measures required for the product.

It answers: What does “complete” mean for this team, this product, this organization?

Why Definition of Done is Critical

Without Definition of Done:

  • Everyone has different quality standards
  • Technical debt accumulates invisibly
  • “90% done” becomes “stuck at 90% forever”
  • Rework surprises the team later
  • Product quality degrades over time

With Clear Definition of Done:

  • Shared quality standards
  • Technical debt is prevented
  • “Done” has objective meaning
  • No surprises at release time
  • Product quality compounds over time

Definition of Done Example (Simple)

For a small web application team:

  • Code written and follows style guide
  • Unit tests written and passing
  • Code reviewed by at least one peer
  • Integrated into main branch
  • No compiler warnings
  • Acceptance criteria met
  • Deployed to staging environment
  • Product Owner has accepted the work

Definition of Done Example (Comprehensive)

For an enterprise team with compliance requirements:

  • Code written following coding standards
  • Unit tests written (minimum 80% code coverage)
  • Unit tests passing
  • Integration tests passing
  • Security scan completed (no high-severity issues)
  • Performance testing completed (meets SLA requirements)
  • Code reviewed and approved by two peers
  • Architecture review completed (for significant changes)
  • Accessibility standards met (WCAG 2.1 AA)
  • Documentation updated:
  • User documentation
  • API documentation
  • Technical architecture docs
  • Deployed to staging environment
  • Smoke tests passing in staging
  • Acceptance criteria verified by Product Owner
  • Compliance checklist completed
  • Release notes updated
  • No known critical or high-priority defects

Definition of Done Example (Startup)

For a fast-moving startup:

  • Code written
  • Basic tests passing
  • Deployed to production
  • Product Owner verified it works
  • No critical bugs

Note: Simpler DoD isn’t “worse” — it matches the context. Startups optimize for speed. Enterprises optimize for safety and compliance.

Creating Your Definition of Done

Step 1: Start with Organizational Standards

If your organization has standards, start there:

  • Security requirements
  • Compliance needs (HIPAA, SOC2, GDPR)
  • Accessibility standards
  • Performance SLAs
  • Documentation requirements

These are non-negotiable minimums.

Step 2: Add Development Best Practices

What engineering practices ensure quality?

  • Code review
  • Automated testing
  • Continuous integration
  • Static analysis
  • Security scanning

Step 3: Include Product Quality Gates

What makes work truly ready for users?

  • Acceptance criteria met
  • Product Owner acceptance
  • User documentation
  • Usability testing (if applicable)

Step 4: Make It Achievable

Can the team actually meet this DoD every Sprint?

If your DoD requires 3 weeks of compliance review, and Sprints are 2 weeks, you have a problem.

Start with what’s achievable. Strengthen over time.

Step 5: Write It Down Explicitly

Make the DoD:

  • Written and visible
  • Specific and unambiguous
  • Checklistable (can verify each item)
  • Agreed upon by the whole team

Step 6: Review and Evolve

The DoD should evolve as the team matures:

Sprint 1 DoD might be:

  • Code written
  • Manual testing completed
  • Deployed to test environment

Sprint 10 DoD might add:

  • Automated unit tests
  • Code review

Sprint 20 DoD might add:

  • Integration tests
  • Security scan
  • Performance testing

Improvement happens gradually.

Living with the Definition of Done

During Sprint Planning

When selecting work, ask: “Can we complete this to our Definition of Done within the Sprint?”

If a story requires compliance review that takes 2 weeks, and your Sprint is 2 weeks, and DoD includes compliance review, you need to either:

  • Change the DoD (remove compliance review)
  • Extend the Sprint
  • Don’t pull that story
  • Do compliance review before Sprint Planning (in refinement)

During the Sprint

Before marking anything “Done,” verify against DoD:

Developer: “I’ve finished the login feature.” Team: “Let’s check the DoD.”

  • Code written? ✓
  • Tests passing? ✓
  • Code reviewed? ✗ (Not yet)
  • Status: Not done. Needs review.**

No shortcuts. No “we’ll do that later.”

During Sprint Review

Only work meeting Definition of Done is demonstrated.

If you have 5 stories, but only 3 meet DoD, you demonstrate 3. The other 2 return to Product Backlog.

No “it’s 90% done” presentations.

During Retrospective

Regularly ask:

  • Is our DoD appropriate?
  • Are we consistently meeting it?
  • Should we strengthen it?
  • Are we using it as a crutch to avoid quality?
  • What’s preventing us from meeting DoD?

Definition of Done vs. Acceptance Criteria

Many teams confuse these. They’re different:

Acceptance Criteria

What: Specific conditions for THIS particular user story.

Example for “User can reset password”:

  • User receives email within 5 minutes
  • Reset link expires after 24 hours
  • New password must meet strength requirements
  • User can log in with new password
  • Old password no longer works

Scope: Story-specific. Changes for every story.

Definition of Done

What: Quality standards for ALL work.

Example:

  • Code reviewed
  • Tests passing
  • Deployed to staging

Scope: Applies to every story, every Sprint.

How They Work Together

Both must be met:

User Story = Done when:

  1. Acceptance criteria met (story is functionally correct)
  2. AND Definition of Done met (quality standards upheld)

Example:

Reset password feature:

  • ✓ Acceptance criteria met (feature works as specified)
  • ✗ Definition of Done not met (no code review yet)
  • Status: Not Done

Common Definition of Done Challenges

Challenge 1: “We Can’t Meet DoD Every Sprint”

Why it happens:

  • DoD is too stringent for Sprint length
  • Team lacks skills/tools
  • Organizational impediments (slow approvals)

Solutions:

  • Simplify DoD to what’s achievable now
  • Invest in automation and tooling
  • Scrum Master escalates organizational impediments
  • Gradually strengthen DoD as capability grows
  • Consider if Sprint length is appropriate

Challenge 2: “Different Items Have Different DoD”

Why it happens:

  • Mix of work types (features, bugs, spikes)
  • Some items have special requirements

Solutions:

  • Create one base DoD that applies to all work
  • Add specific additions for certain types:
  • DoD + (for features: user documentation)
  • DoD + (for bugs: root cause analysis)
  • DoD + (for spikes: written findings document)
  • Keep base DoD consistent

Challenge 3: “Done” Doesn’t Mean “Deployable”

Why it happens:

  • DoD doesn’t include deployment
  • Manual deployment process
  • Organizational policies prevent continuous deployment

Solutions:

  • Add “deployed to staging” to DoD minimum
  • Work toward continuous deployment
  • If production deployment requires approval, that’s fine — but staging deployment should be in DoD
  • “Potentially releasable” means ready to deploy, not necessarily deployed to production

Challenge 4: “We Skip DoD When Under Pressure”

Why it happens:

  • External pressure to “just ship it”
  • Team doesn’t respect DoD
  • DoD seen as optional

Solutions:

  • DoD is not negotiable. If you skip it, you’re not doing Scrum.
  • Make technical debt visible when DoD is violated
  • Track: How many stories met DoD? Should be 100%.
  • Scrum Master protects the team from pressure to violate DoD
  • Product Owner educates stakeholders on consequences

Challenge 5: “DoD Is Too Vague”

Examples of vague DoD:

  • “Good quality”
  • “Properly tested”
  • “Well documented”

Problem: Subjective. Open to interpretation.

Solution: Make it specific and checklistable:

  • ❌ “Properly tested”
  • ✓ “Unit tests passing with >80% coverage”
  • ❌ “Well documented”
  • ✓ “README updated, API endpoints documented in Swagger”

Challenge 6: “DoD Never Changes”

Why it happens:

  • Team forgets DoD should evolve
  • Fear of increasing workload
  • Lack of retrospection on DoD

Solutions:

  • Review DoD every 3–4 months in Retrospective
  • Ask: “Should we strengthen our DoD? Add automation?”
  • Celebrate when team can take on higher standards
  • Frame it as growth, not burden

Technical Debt and Definition of Done

Technical debt is work that violates the Definition of Done but ships anyway.

The Vicious Cycle

Sprint 1: Skip code review to “save time.” Sprint 2: Bug appears in unreviewed code. Costs 2 hours to fix. Sprint 3: Another bug. 3 hours to fix. Sprint 4: Major refactor needed. Costs 2 days.

Total cost: Far more than the 1 hour code review would have cost.

The Virtuous Cycle

Sprint 1: Honor DoD. Code reviewed. Takes extra hour. Sprint 2: No bugs from Sprint 1 code. Team is productive. Sprint 3: Codebase remains clean. Team maintains velocity. Sprint 4: Adding new features is straightforward.

Result: Sustainable pace and quality.

Making Technical Debt Visible

Track violations:

  • How many stories shipped without meeting DoD?
  • What was skipped? (tests, review, documentation)
  • What’s the plan to address it?

If you must violate DoD (rare emergencies):

  • Make it explicit
  • Create backlog item to fix it
  • Prioritize paying it down quickly

Ideally: Never violate DoD. If pressure is that extreme, cancel the Sprint.

Expanding the Definition of Done

As teams mature, DoD typically grows:

Stage 1: Basic DoD (Sprints 1–10)

  • Code written
  • Manual testing
  • Deployed to test environment
  • Product Owner acceptance

Stage 2: Intermediate DoD (Sprints 10–25)

  • Automated unit tests
  • Code review
  • Basic documentation

Stage 3: Advanced DoD (Sprints 25–50)

  • Integration tests
  • Security scanning
  • Performance testing
  • Deployed to staging automatically
  • Comprehensive documentation

Stage 4: Mature DoD (Sprints 50+)

  • Continuous deployment to production
  • Monitoring and alerting configured
  • A/B test framework integrated
  • Accessibility verified
  • All compliance requirements met

This evolution takes time. Be patient. Strengthen gradually.

Definition of Done for Different Contexts

Regulated Industries (Healthcare, Finance)

Must include:

  • Compliance verification
  • Security audits
  • Privacy checks
  • Documentation for auditors
  • Approval gates

DoD might be longer and more rigorous.

Startups

Often simpler:

  • Code works
  • Deployed to production
  • Product Owner verified

Speed over extensive process. Acceptable trade-off in early stage.

Open Source Projects

Might include:

  • Passes all CI checks
  • Documentation updated
  • Changelog entry added
  • Backward compatibility maintained
  • Community review completed

Enterprise Teams

Comprehensive:

  • Multiple testing layers
  • Architecture review
  • Security scan
  • Performance verification
  • Documentation
  • Deployment to staging
  • Smoke tests

The Increment Across Multiple Sprints

Understanding cumulative Increments:

Sprint 1 Increment:

  • User registration
  • Basic login

Sprint 2 Increment:

  • Sprint 1 Increment +
  • Password reset
  • Email verification

Sprint 3 Increment:

  • Sprint 2 Increment +
  • User profile editing
  • Avatar upload

Each Increment is usable. Each builds on the previous.

Key point: You could release after ANY Sprint. That’s what “potentially releasable” means.

Measuring Quality

Lead Indicator Metrics

Definition of Done Compliance: % of stories meeting DoD: Target 100%

Automated Test Coverage: Code coverage percentage: Target >80%

Code Review Completion: % of code reviewed before merge: Target 100%

Lag Indicator Metrics

Defect Escape Rate: Bugs found in production vs. pre-production: Lower is better

Production Incidents: Critical issues per Sprint: Trending down is good

Rework Rate: Stories requiring significant fixes after “Done”: Lower is better

Technical Debt: Time spent on technical debt vs. new features: Should be manageable

Practical Tips

Tip 1: Post DoD Visibly

Physical team: Print Definition of Done poster. Hang near Sprint Backlog.

Remote team: Pin DoD in team channel. Include in Sprint Planning agenda. Reference in Daily Scrum.

Tip 2: Use Checklist

For each story, literally check off DoD items:

Story: Guest Checkout

  • ☐ Code written
  • ☐ Unit tests passing
  • ☐ Code reviewed
  • ☐ Integration tests passing
  • ☐ Deployed to staging
  • ☐ Acceptance criteria met
  • ☐ Product Owner accepted

When all checked: Story is Done.

Tip 3: Automate Verification

Use tools to verify DoD automatically:

  • CI/CD pipeline checks tests
  • Linting checks code style
  • Security scanners run automatically
  • Code coverage reports generated
  • Deployment gates enforce standards

Automation prevents “I forgot to run tests” excuses.

Tip 4: DoD in Sprint Planning

When estimating stories, account for DoD:

Story seems simple (3 points)

  • Writing tests (1 point)
  • Code review time (0.5 points)
  • Documentation (0.5 points) = Actually 5 points

This prevents chronic underestimation.

Tip 5: Demo Only Done Work

In Sprint Review, show only completed work.

If it doesn’t meet DoD, don’t demonstrate it. Be honest about what’s truly done.

This maintains credibility and reinforces DoD importance.

Definition of Done Anti-Patterns

Anti-Pattern 1: The Aspirational DoD

What it looks like: DoD includes items the team never actually does.

Example: “Performance testing completed” (but team never does performance testing)

Why it’s wrong: DoD becomes fiction. Team loses respect for it.

Fix: DoD should reflect reality, not wishes. If you don’t do it, remove it from DoD. Then work to add capability and strengthen DoD later.

Anti-Pattern 2: The “Done Done” Problem

What it looks like: Team has “done” and “done done” (or “dev done” vs. “really done”)

Why it’s wrong: This means you have a weak DoD. “Done” should mean DONE.

Fix: Eliminate “done done.” Strengthen your DoD so “done” actually means complete.

Anti-Pattern 3: The Invisible DoD

What it looks like: DoD exists in a document somewhere, but nobody references it.

Why it’s wrong: If it’s not used, it’s useless.

Fix: Make DoD visible, reference it constantly, use it actively in Sprint Planning and Daily Scrum.

Anti-Pattern 4: The Unchanging DoD

What it looks like: Same DoD for 2 years despite team growth and new tooling.

Why it’s wrong: Missed opportunity to strengthen quality standards as capability grows.

Fix: Review DoD quarterly. As team adds automation, skills, and tools, strengthen DoD.

Anti-Pattern 5: The Negotiable DoD

What it looks like: “Can we skip code review this time? We’re in a hurry.”

Why it’s wrong: DoD is not a suggestion. It’s the quality standard.

Fix: DoD is non-negotiable. If pressure is extreme, cancel Sprint. Don’t compromise quality.

Conclusion

The Increment and Definition of Done are Scrum’s quality insurance. They transform “done” from a vague, subjective concept into a concrete, shared standard.

The Increment:

  • Working software that meets Definition of Done
  • Potentially releasable every Sprint
  • Cumulative — builds on previous Increments
  • The tangible result of the team’s work

The Definition of Done:

  • Explicit quality standard
  • Shared by entire team
  • Non-negotiable
  • Evolves as team matures

When teams honor their Definition of Done:

  • Quality is built in, not bolted on
  • Technical debt is prevented, not accumulated
  • “Done” has objective meaning
  • The product remains healthy Sprint after Sprint
  • Sustainable pace is maintained

When teams violate their Definition of Done:

  • Technical debt compounds
  • Quality degrades
  • Rework multiplies
  • Velocity eventually collapses
  • The product becomes unmaintainable

The choice is stark. Honor your Definition of Done and build quality into every Sprint. Or compromise it and pay the price in technical debt, rework, and declining velocity.

Great teams understand: the fastest way to go fast is to go well. Definition of Done is how you go well.
